## 1. Основная UI/component library

**Рекомендация: shadcn/ui с Base UI в качестве примитивного слоя (не Radix, не MUI/Ant/Chakra/Mantine целиком).**

Контекст, который важно знать, потому что он изменился недавно: исторически shadcn/ui собирался поверх Radix UI. После того как WorkOS купил Radix в 2022 году, темп разработки Radix заметно снизился — часть проблем (Combobox, multi-select) годами оставалась открытой. Часть исходной команды Radix перешла в MUI и с нуля построила **Base UI** — тот же слой headless-примитивов, но активно развивающийся. Base UI получил стабильный релиз 1.0 в декабре 2025, сейчас на версии 1.6, и с июля 2026 shadcn/ui **сделал его дефолтом для новых проектов** (Radix при этом не депрекейтед и продолжает поддерживаться — переключение делается одним флагом). У Base UI сейчас закрыт весь список примитивов, который нужен по брифу: Dialog, Drawer, Popover, Tooltip, Hover Card, Context Menu, Toast, Combobox, Command, Select, Menubar, Navigation Menu, Sidebar, Slider, Progress, Data Table, Chart, Empty-state — то есть почти весь раздел 6 брифа закрывается «из коробки» без написания accessibility-логики самостоятельно.

Почему не полноценная дизайн-система (MUI, Ant Design, Chakra, Mantine) целиком:

- Все они несут готовый визуальный язык (Material, Ant, Chakra-style), из-под которого сложно выйти — а один из ключевых критериев (20% веса) — это визуальная самостоятельность продукта. Снять с MUI «внешность MUI» — отдельная работа, часто более трудоёмкая, чем стилизовать headless-примитив с нуля.
- shadcn/ui устроен иначе: код компонентов копируется в ваш репозиторий, вы им владеете и правите напрямую — то есть это не «ещё одна большая библиотека», а комплект отправных точек. Это соответствует ограничению 14 брифа («не подключать несколько больших библиотек с конфликтующими стилями»).

Почему не голый Radix и не голый Base UI без shadcn-обвязки: shadcn даёт единую CLI, консистентные варианты компонентов и уже готовую интеграцию с Tailwind-темизацией — это экономит время на 3–5-человечной команде, при этом ничего не мешает вам поштучно дописывать/заменять любой компонент.

## 2. Data grid для Player Explorer

**Рекомендация: TanStack Table (headless логика) + TanStack Virtual (виртуализация строк), поверх разметки `shadcn/table` — а не AG Grid и не MUI X Data Grid.**

Аргументы:

1. **Объём данных не требует «тяжёлой артиллерии».** 700–850 строк × 10–20 видимых колонок — это диапазон, где DOM-виртуализация (рендерится только видимая часть + overscan) работает гладко. TanStack Virtual спроектирован именно для такого класса задач и обычно тестируется на десятках-сотнях тысяч строк — 850 для него тривиальны.
2. **TanStack Table закрывает весь список фич из раздела 4.3 «из коробки»**: сортировка (в т.ч. мультиколоночная), глобальный поиск, кастомные функции фильтрации (в т.ч. диапазонные), закрепление колонок, группировка заголовков по смыслу/турам, скрытие/показ колонок, изменение ширины, выбор строк — всё это встроенные модели состояния, а не то, что нужно писать с нуля. Сохранение пользовательских настроек — это просто сериализация объекта состояния таблицы (`columnVisibility`, `columnOrder`, `sorting`, `columnPinning`) в БД/localStorage.
3. **Свобода визуальной кастомизации максимальная** — таблица собирается из ваших же shadcn/Tailwind примитивов, а не перекрашивается поверх чужой темизационной системы. Это напрямую бьёт в критерий с весом 20%.
4. **Лицензионный риск нулевой.** AG Grid Community — это MIT и правда бесплатен, но **row grouping, pivoting, tree data, master-detail, Excel-экспорт — это Enterprise-функции**, требующие лицензии за разработчика (по актуальным на 2026 год прайсам — порядка $1000+/разработчика/год). Сейчас эти функции не нужны, но бриф прямо просит закладывать рост метрик и сценариев (ограничение 14) — есть риск, что через полгода команда упрётся в платную стену, даже не планируя этого.
5. **AG Grid тяжелее стилизовать «не под enterprise-дашборд».** У него собственная тема-система (CSS-переменные AG Grid), которая живёт параллельно вашим Tailwind-токенам — интегрировать её так, чтобы продукт не выглядел как типовая корпоративная админка, требует больше работы, чем кажется на старте, а именно этого бриф просит избежать (раздел 7).

О клавиатурной навигации отдельно — это единственное место, где кажется, что «headless + свой код» проигрывает AG Grid, у которого спредшит-навигация (стрелки между ячейками) есть из коробки. Но здесь стоит явно разделить два паттерна ARIA:

- **`role="grid"`** (спредшит-паттерн, как в Excel/AG Grid) — обязывает реализовать полную 2D-навигацию стрелками, редактирование по Enter/F2 и т.д. Это ровно то, что бриф просит не разрабатывать с нуля (раздел 11).
- **обычная `<table>`** с семантическими `<th>`/`<td>` — по WAI-ARIA Authoring Practices для неё **не требуется** навигация стрелками между ячейками, достаточно обычного Tab-порядка между интерактивными элементами внутри ячеек (кнопка popover, чекбокс выбора и т.д.) плюс видимых focus states.

Player Explorer в брифе — это таблица для просмотра и анализа, не для inline-редактирования как в Excel. Значит, правильный (и более простой) ARIA-паттерн — именно семантическая `<table>`, а не `grid`. Это одновременно закрывает требование «доступная клавиатурная навигация» и не заставляет писать 2D roving-tabindex с нуля. Если позже в продукте появится настоящее inline-редактирование ячеек — тогда имеет смысл вернуться к вопросу и рассмотреть AG Grid Community точечно для этого экрана.

**Когда AG Grid Community стоит держать в уме как fallback:** если сроки поджимают и команда хочет получить закреплённые колонки/сортировку/фильтры без написания собственного слоя над TanStack Table за 1–2 дня. Это честный компромисс скорость/кастомизация, но не основная рекомендация.

## 3. Charts и визуализация данных

**Рекомендация: не менять charting-библиотеку целиком — оставить Recharts для стандартных графиков, добавить точечные кастомные SVG/visx-визуализации для «фирменных» экранов.**

- **Recharts остаётся хорошим выбором** для линий/баров/area-чартов (форма, календарь сложности, распределение прогноза по турам). Он уже в стеке, у него нативная интеграция с shadcn через компонент `Chart` (готовая обвязка под ваши design-токены), SSR-совместим, и для датасетов такого масштаба (десятки точек на график, не тысячи) производительность не является риском.
- **Для действительно уникальных визуализаций** — футбольное поле с расстановкой, радар/лепестковая диаграмма сравнения игроков, индикатор confidence — использовать **кастомный SVG** (поле — это в любом случае не «график» в смысле chart-библиотеки, а координатная схема) либо низкоуровневые примитивы **visx** там, где нужен полный контроль над формой (например, кастомный sparkline с особым дизайном, а не то, что рисует Recharts по умолчанию). Это ровно то различие, которое просит бриф: «меньше ощущения типового AI-дашборда» достигается именно в тех 3–4 визуализациях, которые видны чаще всего и определяют лицо продукта — остальные графики могут спокойно быть Recharts.
- Recharts и visx используют SVG и не конфликтуют между собой технически — оба спокойно живут в одном дереве компонентов.

## 5. Нужен ли Tailwind после перехода

**Да, однозначно оставить Tailwind CSS 4.** Base UI сам по себе стилистически нейтрален и не требует никакого конкретного CSS-подхода — но shadcn/ui как обвязка вокруг него **написан на Tailwind-утилитах**, и это не тот случай, когда стоит бороться с инструментом. Для команды из 3–5 человек Tailwind — это по факту ваша дизайн-система: единый источник токенов (цвет, отступы, типографика, радиусы), без необходимости поддерживать отдельный CSS-in-JS слой или ручные CSS-модули. Отказ от Tailwind означал бы переписывать стилизацию всех shadcn-компонентов вручную — лишняя работа без выгоды.

## 6. Design tokens и темизация

Организовать через **Tailwind v4 `@theme` + CSS custom properties**, по той же модели, что использует сам shadcn:

- Токены объявляются как CSS-переменные (`--color-*`, `--radius-*`, `--font-*`) в одном месте, Tailwind-утилиты и все компоненты ссылаются на них — смена темы (light/dark) — это переключение значений переменных на уровне `<html data-theme>`, без изменения компонентов.
- **Именование токенов — семантическое, а не «визуальное».** Не `--blue-500`, а `--color-forecast`, `--color-fact`, `--color-risk`, `--color-positive-delta`, `--color-negative-delta`. Это прямое требование раздела 7 брифа («чёткое различие между фактом, прогнозом и неопределённостью») — если цвет привязан к смыслу на уровне токена, а не подбирается на каждом экране заново, разница между «факт / прогноз / неопределённость» будет физически невозможно нарушить случайно.
- Отдельная группа токенов для confidence/uncertainty (например, opacity- или texture-based модификатор поверх базового цвета, а не отдельная палитра) — так прогноз с низкой уверенностью визуально «менее уверен», а не просто другого цвета.
- Плотность таблиц (compact/comfortable) — тоже токенизируется (`--row-height-compact`, `--row-height-comfortable`), а не хардкодится в компоненте.

## 7. Как обеспечить визуальную уникальность

Бриф прямо перечисляет, чего избегать: тёмно-синий/slate/emerald/teal-дуэт, сине-зелёные градиенты, десятки одинаковых карточек, чрезмерные скругления, свечения, вид крипто-панели или букмекерской конторы. Это описание «дефолтного AI-сгенерированного SaaS» очень точное, и от него действительно стоит осознанно уходить. Несколько конкретных, проверяемых принципов (не финальная палитра — это отдельная дизайнерская задача, но направление):

1. **Один акцентный цвет, а не палитра из пяти.** Остальной интерфейс — нейтральные тона (бумажный светлый / чернильный тёмный), акцент используется дозированно — на первичных действиях и на действительно важных сигналах.
2. **Цвет закреплён за смыслом, не за декором.** Если зелёный используется для «хорошего прогноза», он не может параллельно использоваться как акцент кнопки — иначе смысловая нагрузка размывается.
3. **Меньше карточек, больше разделителей.** Вместо десятков bordered/shadowed карточек на дашборде — таблично-подобная структура со строгими границами (border, не shadow), как в хорошей спортивной статистике (Opta/FBref-стиль), а не в SaaS-дашборде.
4. **Сдержанные радиусы** (2–6px, не 16–24px) — это то, что сильнее всего выдаёт «типовой AI-дашборд» на глаз.
5. **Типографика с характером в цифрах.** Для таблиц и метрик — шрифт с настоящими табличными (моноширинными) цифрами, чтобы колонки чисел визуально выравнивались (например, Inter, IBM Plex Sans или Public Sans — все три открытые, с хорошими tabular figures). Для заголовков можно взять более выразительную гарнитуру, отличную от тела текста — это дёшево и сильно снижает ощущение шаблонности.
6. **Свой «фирменный» визуальный элемент** — расстановка на поле. Это единственный экран, где имеет смысл вложиться в действительно кастомную графику (не просто раскрашенный SVG-прямоугольник, а продуманная схема с типографикой имён, статус-иконками), потому что он не встречается в типовых дашбордах и сразу считывается как «сделано под конкретный продукт».

## 8. Совместимость с Next.js 15 / React 19 / SSR

И Base UI, и Radix заявлены как совместимые с React 17/18/19, лицензированы MIT, и оба — клиентские библиотеки (используют `useState`/порталы/обработчики событий), то есть компоненты, которые их используют, обязаны быть `"use client"`. Практический паттерн для App Router:

- Держите Server Components на уровне страниц/layout и там, где идёт получение данных из Prisma/Postgres (Dashboard-сводка, состав из БД, список игроков) — это основная часть раздела 2 брифа «использовать Server Components там, где возможно».
- Интерактивные листья дерева (сама таблица, комбобоксы фильтров, диалоги, popovers, футбольное поле с интерактивностью) — клиентские компоненты, которые получают уже готовые данные пропсами с сервера.
- Гидратация не должна течь, если избегать общих ошибок: рандом/`Date.now()` в рендере без синхронизации сервер/клиент, различия в форматировании чисел/дат между Node и браузером (стоит зафиксировать locale явно, а не полагаться на системный) — это стандартные грабли Next.js, не специфичные для Base UI.
- 850 строк данных для таблицы стоит грузить как Server Component (fetch из Prisma) и передавать в клиентский TanStack Table как начальные данные — тяжёлый JSON не обязан идти через client fetch.

Существенных технических рисков совместимости с вашим стеком не видно.

## 9. Какие функции окажутся платными

**Ни одна функция из брифа не требует платной лицензии** при рекомендуемом стеке (Base UI/Radix, shadcn, Tailwind, TanStack Table/Virtual, Recharts, visx — всё MIT или Apache 2.0, бесплатно для коммерческого использования без ограничений по местам/пользователям).

Единственная точка, где деньги потенциально могли бы появиться — это AG Grid Enterprise, если бы вы позже захотели row grouping / pivot-таблицы / tree data / master-detail / встроенный Excel-экспорт из грида. По актуальным на 2026 год ценам это лицензия на уровне разработчика (порядка $1000–2000/разработчика/год в зависимости от типа лицензии) — но поскольку рекомендация — не использовать AG Grid, этот сценарий сейчас не актуален. Если такая функциональность понадобится в будущем именно в виде «пивот-таблицы» — это единственный класс задач, где стоит рассмотреть точечное подключение AG Grid Enterprise именно на одном экране, а не менять весь grid-слой продукта.

## 10. Риски производительности и доступности

**Производительность:**

- Главный риск — не количество строк (850 — немного), а **насыщенность ячеек** (popover с разбивкой прогноза, sparkline, несколько иконок статуса в каждой строке). Виртуализация обязательна не столько из-за объёма данных, сколько из-за веса каждой строки при полном рендере.
- При закреплённых колонках + виртуализации нужно внимательно синхронизировать скролл между «закреплённой» и «прокручиваемой» частью таблицы — это самое трудоёмкое место интеграции TanStack Table + TanStack Virtual, стоит заложить на это отдельное время в оценке.
- Открытие popover внутри виртуализированной строки, которая может «уехать» при скролле — нужно либо закрывать popover при скролле контейнера, либо рендерить его через портал, не привязанный к позиции строки в DOM.
- Recharts на десятках графиков одновременно (например, в Player Explorer, если добавить мини-графики в каждую строку) может начать давать просадки — для sparklines внутри таблицы лучше использовать лёгкий кастомный SVG, а не полноценный компонент Recharts на каждую строку.

**Доступность:**

- Виртуализация строк ломает наивную семантику `<table>`, если не добавить `aria-rowcount`/`aria-rowindex`/`aria-setsize` — иначе скринридер не поймёт, что видит окно из большего списка. Это нужно закладывать в реализацию грида с самого начала, а не добавлять постфактум.
- Требование брифа «цвет — не единственный индикатор» (раздел 9) нужно проверять по каждому статус-индикатору отдельно: тренд, confidence, доступность игрока — у каждого должен быть неprivethbcolor-сигнал (иконка направления, форма, текст), а не только заливка.
- Compare tray и любые модальные потоки — фокус должен корректно перемещаться внутрь при открытии и возвращаться к триггеру при закрытии; Base UI это делает автоматически для Dialog/Drawer/Popover, но для кастомных элементов (например, самого compare tray, если он не построен на `Sheet`) это нужно реализовать вручную.
- `prefers-reduced-motion` — учесть на уровне токенов анимации (одна переменная длительности transition, которая обнуляется в reduced-motion медиа-запросе), а не на каждом компоненте отдельно.
- Карточный режим таблицы на мобильном — легко забыть, что он тоже должен быть доступен с клавиатуры/скринридера, а не только тач-френдли.

## 11. Миграция с текущего интерфейса

Поскольку shadcn/ui уже не обязателен к сохранению, а Radix/Base UI сосуществуют в рамках одной кодовой базы (shadcn поддерживает оба одновременно, с флагом на уровне `components.json`), миграция может идти поэтапно, без «бигбэнга»:

1. **Фаза 0 — фундамент.** Завести токены (цвет/типографика/радиусы/плотность), настроить `shadcn init` на Base UI, определить light/dark. Ничего не переносится из старого UI — это инфраструктура.
2. **Фаза 1 — каркас.** Navigation shell, header, layout. Маленькая поверхность, но задействует почти все базовые примитивы (Sidebar, Tabs, Avatar, Badge) — хороший способ проверить токены на практике до того, как строить сложные экраны.
3. **Фаза 2 — Player Explorer.** Самый рискованный и самый тяжёлый по весу критериев экран (25% — функциональность data grid). Сделать его вторым, а не последним, чтобы все сложные технические решения (виртуализация, закрепление колонок, паттерн доступности таблицы, filter chips) были обкатаны рано и переиспользованы на остальных экранах.
4. **Фаза 3 — My Team.** Включает самый визуально уникальный компонент (поле) — имеет смысл делать его после того, как токены и общий визуальный язык устоялись на Фазе 1–2, иначе поле придётся переделывать вместе с ребрендингом.
5. **Фаза 4 — Player Details, Comparison, Transfer Advisor.** Переиспользуют карточку игрока и confidence-индикаторы, построенные в Фазе 2–3.
6. **Фаза 5 — AI Assistant.** Готовые `Message`/`Bubble`/`Attachment`-компоненты снижают объём работы — ставить не первым, а после того как структурированные представления игрока/сравнения уже существуют (ассистент их переиспользует, а не изобретает заново).
7. **Фаза 6 — Settings/Onboarding, Data Status.** Технически самые простые экраны, наименьший риск — логично оставить на конец.

Поскольку продукт используют 3–5 человек и это не публичный сервис, старый и новый UI могут какое-то время сосуществовать по роутам (`/team` — новое, `/predictions` — ещё старое) без необходимости синхронного релиза всего сразу.

## 12. С какого экрана начинать

**Player Explorer.** Не Dashboard и не My Team, при всей их важности как «лица продукта» — по двум причинам:

- Это самый тяжёлый по весу критерий брифа (data grid — 25% итоговой оценки) и самый большой источник технического риска (виртуализация, закрепление колонок, доступность таблицы). Лучше столкнуться с этими решениями в начале проекта, когда их ещё дёшево пересмотреть, чем в середине, когда уже построено 4 экрана поверх неверного решения.
- Он вынуждает рано принять решения, которые понадобятся везде: как выглядит badge статуса, как выглядит confidence-индикатор, как оформлена ячейка с прогнозом и её popover — то есть Player Explorer на практике становится черновиком дизайн-системы, а не просто одним экраном.

Единственное, что стоит сделать до него — минимальный каркас из Фазы 0–1 (токены + layout shell), чтобы таблица сразу собиралась на правильных примитивах, а не переделывалась потом.

## Итог: что делать в первые две недели

1. `npx shadcn init` с Base UI, поднять токены (цвет/радиус/типографика/плотность) на Tailwind 4 `@theme`.
2. Собрать navigation shell + light/dark на этих токенах.
3. Прототип Player Explorer: TanStack Table + TanStack Virtual на реальных ~800 строках тестовых данных, семантическая `<table>` (не ARIA grid), закрепление 2 колонок, popover в ячейке прогноза.
4. Замерить производительность прокрутки/фильтрации на этом прототипе до того, как тиражировать паттерн на остальные экраны.
